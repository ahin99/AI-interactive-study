# LexiRecall 최종 수정사항 설계문서

## 1. 문서 목적과 구현 범위

이 문서는 `planning/2차_수정사항.md`를 기준으로, 현재 `lexirecall` 앱을 최종 시연 요구사항에 맞게 수정하기 위한 구현 설계 문서다. Claude Code CLI가 이 문서만 보고 바로 구현을 시작할 수 있도록 현재 코드 분석, 목표 사용자 흐름, 데이터 구조, 컴포넌트 변경 계획, 파일별 변경 계획, 구현 순서, 완료 기준을 정리한다.

이번 구현 범위는 고정된 시연용 mock 데이터 기반의 프론트엔드 MVP 수정이다. 실제 AI API 연결, 실제 PDF 파싱, 실제 DB 저장, 정교한 메타인지 계산 공식, AI 프롬프트 설계는 포함하지 않는다.

반드시 유지할 제한:

- 학습 단위는 파일이 아니라 주차다.
- 한 주차에 여러 파일을 업로드할 수 있다.
- 같은 주차의 여러 파일은 하나의 주차 개념 지도로 통합한다.
- 백지복습이 서비스의 핵심 흐름이다.
- AI는 정답과 모범답안을 제공하지 않고 생각할 단서만 제공한다.
- 현재 단계에서는 `materials` 폴더의 시연 자료와 mock 데이터를 사용한다.
- 지식 지도와 메타인지 지도는 같은 타일 구조를 사용하고 색상만 다르게 표현한다.
- 애플리케이션 코드는 이 문서 작성 단계에서 수정하지 않는다.

## 2. 서비스 핵심 원칙

LexiRecall은 법학 개념을 더 쉽게 설명해 주는 서비스가 아니라, 학생이 백지 상태에서 직접 설명하며 자기 이해 수준을 확인하게 만드는 서비스다.

핵심 원칙:

- 자료 업로드는 출발점이고, 학습의 중심은 주차 단위 백지복습이다.
- 개념 지도는 학습자가 무엇을 알고 있다고 느끼는지, 실제로 무엇을 설명할 수 있는지 비교하기 위한 시각 도구다.
- 피드백은 정답 제공이 아니라 누락, 혼동, 구분 기준, 다시 확인할 쟁점을 알려주는 최소 단서다.
- 메타인지 지도는 "안다고 생각했지만 실제로 충분히 설명하지 못한 지점"을 보여준다.
- 사용자는 과목을 선택하고, 주차를 선택하고, 사전 자기평가 후 백지복습을 수행하며, 피드백과 이해 검증 질문에 답한 뒤 지도 변화를 확인한다.

## 4. 현재 구현과 목표 구현의 차이

핵심 차이:

| 영역 | 현재 구현 | 목표 구현 |
| --- | --- | --- |
| 학습 단위 | `materialId`, `conceptId` 중심 | `subjectId`, `weekId` 중심 |
| 자료 구조 | 한 자료가 한 주차처럼 사용됨 | 한 주차에 여러 업로드 파일 가능 |
| 지도 구조 | 개념 리스트 또는 트리 | 전체 화면은 주차 큰 타일, 주차 확대 시 개념 조각 타일 |
| 메인 화면 | 서비스 소개와 최근 자료 | 과목 카드형 대시보드 |
| 과목 상세 | 없음 | 과목별 지식/메타인지 지도, 주차 확대, 기록 확인 |
| 백지복습 대상 | 선택한 단일 개념 | 선택한 주차 전체 |
| 자기평가 | 트리 개념 체크 | 주차 확대 화면에서 해당 주차 개념 체크 |
| 난이도 | 1단계 목차+키워드, 2단계 목차, 3단계 백지 | 목차 전체, 핵심 키워드 3개, 완전 백지 |
| 피드백 | 목록형 피드백 | 학생 답안 블록/문장 기반 형광펜 첨삭 + 별도 이해 검증 질문 입력 |
| 이해 검증 질문 | 질문 목록만 표시 | 질문 3개와 학생 답변 입력, 정답 공개 없음 |
| 결과 저장 | 마지막 피드백 1개 중심 | 복습 세션 기록 누적, 지도 snapshot 갱신 |
| 지식 지도 색 | complete/partial/misconception/missing 4단계 | 회색/빨강/주황/노랑/초록 5상태 |
| 메타인지 지도 | 계산 결과와 보라 ring | 0-100 mock gap 값 기반 핑크 타일 |
| 지도 변화 | 없음 | 3-5개 snapshot 슬라이더 |

2차 문서에는 보라색 상태가 언급되지만, 3차 문서의 정리 규칙을 최종 기준으로 사용한다. 따라서 백지 답안은 정확하지만 이해 검증 답변이 부족한 경우는 보라색이 아니라 노랑으로 처리한다.

## 5. 최종 사용자 흐름

최종 흐름:

1. 메인 대시보드에서 과목 카드를 확인한다.
2. 과목 카드를 클릭해 과목 상세 화면으로 들어간다.
3. 과목 상세에서 지식 지도 또는 메타인지 지도를 선택한다.
4. 전체 지도에서 주차 큰 타일을 클릭한다.
5. 주차가 확대되고, 주차 안의 개념 조각 타일이 글자와 함께 표시된다.
6. 사용자는 해당 주차의 복습 횟수, 이전 지도, 이전 복습 기록을 확인할 수 있다.
7. `백지복습 하기`를 누른다.
8. 해당 주차 개념에 대해 사전 자기평가를 체크한다.
9. 백지복습 난이도를 선택한다.
10. 주차 단위 백지 답안을 작성한다.
11. 첨삭형 피드백을 확인한다.
12. 소크라테스식 이해 검증 질문 3개에 답한다.
13. 정답 공개 없이 결과를 저장한다.
14. 지식 지도와 메타인지 지도가 갱신된다.
15. 과목 상세에서 snapshot 슬라이더로 이전-이후 변화를 확인한다.

## 6. 페이지 및 라우팅 설계

현재 라우트를 최대한 재사용하되, 과목 상세와 주차 중심 구조가 필요하므로 신규 동적 라우트를 추가한다.

| 라우트 | 현재 여부 | 역할 |
| --- | --- | --- |
| `/` | 기존 | 메인 과목 대시보드로 전면 수정 |
| `/upload` | 기존 | 주차 단위 다중 파일 업로드 mock 화면으로 수정 |
| `/subjects/[subjectId]` | 신규 | 과목 상세 화면. 지식/메타인지 지도, 주차 확대, snapshot 슬라이더 |
| `/subjects/[subjectId]/weeks/[weekId]/recall` | 신규 | 주차 단위 자기평가, 난이도 선택, 백지 답안 작성 |
| `/subjects/[subjectId]/weeks/[weekId]/feedback` | 신규 | 첨삭형 피드백, 이해 검증 질문 답변, 결과 저장 |
| `/concept-map` | 기존 | 최종 흐름에서는 제거하지 않고 `/subjects/civil-law-general`로 안내하거나 호환용으로 간소화 |
| `/recall` | 기존 | 최종 흐름에서는 신규 recall 라우트로 리다이렉트하거나 데모 호환용으로 유지 |
| `/feedback` | 기존 | 최종 흐름에서는 신규 feedback 라우트로 리다이렉트하거나 데모 호환용으로 유지 |
| `/knowledge-map` | 기존 | 최종 흐름에서는 과목 상세로 통합하거나 `/subjects/civil-law-general`로 안내 |

라우트 설계 원칙:

- 핵심 시연은 `/` -> `/subjects/civil-law-general` -> `/subjects/civil-law-general/weeks/week-4/recall` -> `/subjects/civil-law-general/weeks/week-4/feedback`로 연결한다.
- 기존 라우트 파일은 삭제하지 않는다. 필요하면 기존 페이지를 새 흐름으로 연결하는 얇은 안내/리다이렉트 성격으로 수정한다.
- 실제 Next 리다이렉트 대신 UI 버튼으로 이동해도 시연상 충분하다.

## 7. 데이터 모델과 상태 구조

현재 `types.ts`는 자료와 개념이 중심이다. 주차 중심 모델을 추가해야 한다.

권장 타입:

```ts
export type KnowledgeStatus =
  | "unreviewed"
  | "not_recalled"
  | "misconception"
  | "partial"
  | "complete";

export type MapMode = "knowledge" | "metacognition";

export type RecallDifficulty = "outline" | "keywords" | "blank";

export interface Subject {
  id: string;
  name: string;
  completionRate: number;
  metacognitionGapRate: number;
  lastReviewedAt: string;
  lastReviewHint: string;
  materialCount: number;
  recent: boolean;
}

export interface StudyWeek {
  id: string;
  subjectId: string;
  weekNumber: number;
  title: string;
  materialIds: string[];
  reviewCount: number;
}

export interface WeekMaterial {
  id: string;
  subjectId: string;
  weekId: string;
  displayName: string;
  fileName: string;
  uploadedAt: string;
  status: "ready" | "analyzing" | "analyzed" | "failed";
}

export interface WeekConcept {
  id: string;
  weekId: string;
  title: string;
  type: ConceptType;
  description?: string;
  outlineLabel?: string;
  keyword?: string;
}

export interface ConceptMapTile {
  conceptId: string;
  status: KnowledgeStatus;
  metacognitionGap: number | null;
}

export interface MapSnapshot {
  id: string;
  subjectId: string;
  weekId?: string;
  label: string;
  reviewNumber: number;
  createdAt: string;
  tilesByConceptId: Record<string, ConceptMapTile>;
}

export interface RecallRecord {
  id: string;
  subjectId: string;
  weekId: string;
  difficulty: RecallDifficulty;
  selfAssessments: SelfAssessment[];
  answerText: string;
  verificationAnswers: Record<string, string>;
  feedbackId: string;
  submittedAt: string;
}

export interface AnnotatedAnswerBlock {
  id: string;
  text: string;
  highlights: {
    phrase: string;
    status: "misconception" | "partial" | "complete";
    hint?: string;
  }[];
}

export interface WeekFeedbackResult {
  id: string;
  recallRecordId: string;
  answerBlocks: AnnotatedAnswerBlock[];
  missingHints: string[];
  verificationQuestions: {
    id: string;
    type: "reason" | "boundary" | "comparison" | "application" | "premise";
    question: string;
  }[];
  nextReviewHint: string;
  statusByConceptId: Record<string, KnowledgeStatus>;
}
```

상태 구조 변경:

- `selectedMaterialId`는 핵심 상태에서 제거하거나 보조 상태로 낮춘다.
- `selectedSubjectId`, `selectedWeekId`, `selectedMapMode`, `selectedSnapshotIndex`를 추가한다.
- `selectedConceptId` 중심의 복습 흐름은 `selectedWeekId` 중심으로 바꾼다.
- `recallAnswer`, `recallDifficulty`, `verificationAnswers`, `currentFeedback`, `recallRecords`, `mapSnapshots`를 store에 둔다.
- persist는 계속 사용한다. 데모 반복을 위해 `resetCurrentRecall`, `saveRecallRecord`, `selectWeek`, `setMapMode`, `setSnapshotIndex`, `toggleWeekSelfAssessment` 액션을 제공한다.

## 11. 지식 지도와 메타인지 지도 설계

관련 파일:

- 수정 또는 대체: `lexirecall/src/components/knowledge-map.tsx`
- 신규 권장: `lexirecall/src/components/week-mosaic-map.tsx`

지식 지도 상태:

| 상태 | 의미 | 색상 |
| --- | --- | --- |
| `unreviewed` | 미복습 | 회색 |
| `not_recalled` | 핵심 내용을 전혀 작성하지 못함 | 빨강 |
| `misconception` | 오개념 포함 | 주황 |
| `partial` | 일부 핵심 내용 누락 또는 이해 검증 답변 부족 | 노랑 |
| `complete` | 정확히 설명함 | 초록 |

메타인지 지도 상태:

| 값 | 의미 | 색상 |
| --- | --- | --- |
| `null` | 미측정 | 회색 |
| `1-40` | 과대평가 오차 작음 | 옅은 핑크 |
| `41-70` | 과대평가 오차 중간 | 중간 핑크 |
| `71-100` | 과대평가 오차 큼 | 짙은 핑크 |

지도 구조:

- 전체 지도에서는 큰 타일 하나가 주차다.
- 각 주차 타일 내부에는 개념 조각을 작은 색상 칸으로만 보여준다.
- 전체 지도에서는 개념명 텍스트를 표시하지 않는다.
- 주차 확대 상태에서는 개념 조각 타일을 동일 크기로 표시하고 개념명을 포함한다.
- 지식 지도와 메타인지 지도는 같은 component와 같은 데이터 순서를 사용한다.
- 색상 계산 함수만 `mode`에 따라 다르게 적용한다.

사용자 행동:

- 전체 주차 타일 hover: 주차 번호, 복습 횟수, 완성률 정도의 짧은 정보 표시
- 주차 클릭: 확대
- 확대 타일 클릭: 해당 개념의 짧은 단서 또는 상태 설명 표시 가능. 단, 정답 설명은 금지

완료 확인:

- 모드만 바꿔도 타일 위치와 크기가 변하지 않는다.
- 미복습 개념은 지식 지도에서 회색이다.
- 메타인지 미측정 개념도 회색이다.

## 19. 디자인 방향

전체 방향:

- 전문적인 법학 학습 도구 느낌
- 데스크톱 우선
- 흰색 또는 매우 옅은 회색 배경
- 지도 영역을 가장 큰 시각적 영역으로 배치
- 카드 사이 간격 충분히 확보
- 일반 UI 색상은 절제하고, 상태 색상은 지도와 피드백에 집중
- 핵심 지표는 큰 숫자로 표시
- 텍스트 과밀 방지
- 버튼과 카드 모서리는 중간 정도 둥글게 유지

구체 지침:

- 과목 대시보드는 실제 폴더 모양이 아니라 과목별 카드 묶음이다.
- 지도 타일은 동일 크기 조각을 사용한다.
- 주차 확대 시 개념명 텍스트가 타일 안에서 잘리지 않도록 최소 높이와 반응형 grid를 지정한다.
- 모바일도 깨지지 않게 하되, 최적 UX는 데스크톱 기준으로 설계한다.
- 기존 teal/purple 과다 사용은 줄이고, 상태 색상과 구분한다.

## 23. 테스트 및 완료 기준

필수 테스트:

- `npm run lint`
- `npm run build`
- 브라우저 수동 확인

수동 확인 시나리오:

1. `/` 접속
2. 민법총칙 과목 카드 정보 확인
3. 과목 카드 클릭
4. 지식 지도와 메타인지 지도 전환
5. 4주차 타일 클릭
6. 주차 확대 지도 확인
7. snapshot 슬라이더 이동
8. 백지복습 하기 클릭
9. 자기평가 체크
10. 난이도 `핵심 키워드 3개` 선택
11. 예시 답안 입력
12. 제출
13. 첨삭형 피드백과 질문 3개 확인
14. 질문 답변 입력
15. 결과 저장
16. 과목 상세로 돌아와 주차 복습 횟수, 지도 색, 기록 갱신 확인

완료 기준:

- 학습 단위가 화면과 상태 모두 주차 중심이다.
- 한 주차에 여러 파일이 연결될 수 있다.
- 전체 지도에서는 주차 단위로 보이고, 주차 확대에서는 개념 조각이 보인다.
- 지식 지도와 메타인지 지도는 같은 타일 구조다.
- 지식 지도 색상은 회색/빨강/주황/노랑/초록 규칙을 따른다.
- 메타인지 지도는 회색/핑크 계열 규칙을 따른다.
- 백지복습 난이도는 목차 전체/핵심 키워드 3개/완전 백지다.
- 피드백은 정답을 제공하지 않는다.
- 이해 검증 질문 3개에 답할 수 있다.
- 결과 저장 후 지도와 기록이 갱신된다.
- 실제 AI API, 실제 PDF 파싱, 정교한 계산식이 들어가지 않는다.

## 24. 구현 전 확인 사항

아래 항목은 구현자가 임의로 확정하지 말고, 필요 시 사용자에게 확인하거나 합리적 기본안으로 명시하고 진행한다.

- 2차 문서에는 보라색 상태가 언급되지만 3차 문서의 최종 규칙은 보라색을 제외한다. 기본안은 "이해 검증 답변 부족은 노랑"이다.
- 메타인지 오차율 대시보드 예시가 `92%`로 매우 높다. 기본안은 사용자 요구 예시를 유지하되, 개념별 gap mock 값의 평균과 정확히 맞추지는 않는다.
- 과목 추가 버튼은 실제 과목 생성 기능까지 구현할지, 현재는 시연용 비활성/안내 버튼으로 둘지 확인이 필요하다. 기본안은 비활성 mock 버튼이다.
- 업로드 화면에서 사용자가 "파일명"을 직접 작성한다고 했는데, 브라우저 파일 선택의 실제 파일명과 별도 표시명을 둘지 확인이 필요하다. 기본안은 `displayName` 입력과 실제 `fileName`을 모두 저장한다.
- 기존 `/concept-map`, `/recall`, `/feedback`, `/knowledge-map` 라우트를 완전히 새 흐름으로 대체할지, 호환 안내 화면으로 남길지 확인이 필요하다. 기본안은 삭제하지 않고 새 흐름으로 안내한다.
- 이전 지도 보기는 전체 과목 snapshot과 주차별 snapshot을 모두 요구한다. 기본안은 하나의 snapshot 데이터에서 `weekId` 필터로 주차별 보기를 제공한다.

## 최종 검토 체크리스트

- 현재 코드 구조와 실제 파일 경로를 반영했다.
- `planning/2차_수정사항.md`의 주차 중심 요구, 지도 요구, 피드백 요구를 반영했다.
- 3차 문서의 최종 규칙에 따라 지식 지도 색상과 메타인지 지도 규칙을 정리했다.
- 사용자 흐름이 과목 선택부터 결과 저장과 지도 갱신까지 연결된다.
- 실제 AI API, AI 프롬프트, 실제 PDF 파싱은 범위에서 제외했다.
- 존재하지 않는 파일은 모두 신규로 표시했다.
- 앱 코드를 수정하지 않고 설계 문서만 작성하는 작업 범위에 맞다.
